# PRD - 微信公众号多账号配置（house-ai）

> 文档类型：简单 PRD（默认模式）

> 作者：产品经理 许清楚（Xu）
> 日期：2026-07-05
> 关联代码：`backend/services/wechat_service.py`、`backend/routes/publish.py`、`backend/config.py`、`frontend-vue/src/pages/PublishPage.vue`

## 0. 项目信息

| 项 | 内容 |
| --- | --- |
| 原始需求 | 支持配置**多个微信公众号账号**，提供配置页面让用户填写/管理账号；发布文章到草稿箱时可**选择目标账号** |
| 技术栈 | 后端 FastAPI + Python（SQLAlchemy async + aiosqlite）；前端 Vue3 + Element Plus |
| 模式 | **增量功能**，不推翻现有单账号实现，而是将其扩展为多账号能力 |
| 关键约束 | AppSecret 属敏感凭证，配置页面**不得明文暴露给前端**，必须由后端加密存储 |

---

## 1. 产品目标

在房屋租赁 AI 营销系统现有单账号微信公众号草稿箱能力（见 `backend/services/wechat_service.py` 的 `WechatService.create_draft`）基础上，扩展为支持**同一主体下多个公众号**的统一管理：提供可视化配置页面让运营人员新增/编辑/删除公众号账号（账号名称、AppID、AppSecret、备注），在发布草稿时可由用户选择目标账号。目标是让一个系统服务多个房源矩阵号/城市号的运营场景，提升多账号分发效率，同时将 AppSecret 等敏感凭证严格收拢在后端加密存储，杜绝前端明文泄露风险。

## 2. 用户故事

| 编号 | 用户故事 |
| --- | --- |
| US-1 | 作为**运营人员**，我希望在配置页面填写多个公众号的 AppID/AppSecret 并保存，以便不需要改代码或 .env 即可接入新账号。 |
| US-2 | 作为**运营人员**，我希望在账号列表中看到所有已配置的公众号及其状态，以便快速核对哪个账号已启用、哪个是默认账号。 |
| US-3 | 作为**运营人员**，我希望发布草稿时能从一个下拉框选择要发往哪个公众号，以便把不同房源内容精准投到对应矩阵号。 |
| US-4 | 作为**运营人员**，我希望能编辑或删除某个账号，以便账号停用/转让时及时清理，避免误发到失效账号。 |
| US-5 | 作为**系统管理员**，我希望 AppSecret 在前端永远不以明文展示/回传，以便符合凭证安全与合规要求。 |

## 3. 需求池

> 优先级：P0 必须 / P1 重要 / P2 可选。每条含编号、描述、验收标准。

### P0（必须）

| 编号 | 描述 | 验收标准 |
| --- | --- | --- |
| R-01 | **多账号数据模型**：新增账号存储实体，至少含 `name`（账号名称）、`app_id`（AppID，唯一）、`app_secret`（密文）、`remark`（备注）、`is_active`（启用/禁用）、`is_default`（默认账号）、时间戳。 | 数据库中存在对应表（建议 `wechat_accounts`，见待确认 Q1）；`app_id` 唯一索引；插入/查询接口可用。 |
| R-02 | **后端账号 CRUD 接口**：提供账号的增/删/改/查 REST 接口（如 `/wechat-accounts`）。 | 新增、列表、编辑、删除四个接口均可用；列表接口**返回 AppSecret 脱敏/不返回明文**。 |
| R-03 | **AppSecret 加密存储**：账号入库时 AppSecret 必须由后端加密，明文不落库、不回传前端。 | 数据库中 `app_secret` 字段为密文；解密仅发生在后端调用微信 API 前；前端任何接口响应均不含明文。 |
| R-04 | **发布接口支持指定账号**：`/publish/wechat` 增加目标账号参数（建议 `wechat_account_id`）。 | 请求携带 `wechat_account_id` 时，使用该账号凭证调用 `WechatService.create_draft`；缺省时回落到默认账号（见 R-06）。草稿创建成功/失败逻辑与现有 `PublishLog` 记录保持一致。 |
| R-05 | **`WechatService` 多账号化改造**：打破现有全局单例直接读 `settings.WECHAT_APPID/APPSECRET` 的写法，使 `create_draft` 能按"账号"维度工作。 | 支持按账号构造服务实例/传入凭证；`access_token` 缓存按 `appid` 维度隔离，避免多账号互相覆盖（当前为实例级缓存）。现有单账号路径向后兼容。 |
| R-06 | **默认账号机制**：系统支持标记一个默认账号，发布未指定账号时使用它。 | 仅一个账号为 `is_default=true`；列表/发布回落正确。 |

### P1（重要）

| 编号 | 描述 | 验收标准 |
| --- | --- | --- |
| R-07 | **配置页面 UI**：独立「公众号配置」页面，含账号列表 + 新增/编辑表单。 | 列表展示账号名称、AppID（脱敏）、备注、状态；表单字段含账号名称、AppID、AppSecret（密码框）、备注；保存/编辑/删除可触发对应接口。 |
| R-08 | **发布页「选择公众号」下拉**：在 `PublishPage.vue` 选择"微信公众号"平台时，展示账号下拉框。 | 下拉拉取 `/wechat-accounts` 列表（仅启用账号）；选中后随发布请求带上 `wechat_account_id`；未选且有默认账号时显示默认提示。 |
| R-09 | **账号启用/禁用**：支持停用账号。 | 禁用的账号不出现在发布下拉中；若已禁用账号被显式指定发布，后端返回明确错误。 |
| R-10 | **AppID 校验与去重**：新增/编辑时校验 AppID 格式与唯一性（同一主体下不同账号 AppID 必不同）。 | 重复 AppID 或非法格式被拒绝并返回友好提示。 |
| R-11 | **发布记录关联账号**：`PublishLog` 记录所发往的账号（建议新增 `wechat_account_id` 列）。 | 日志可追溯到具体公众号；历史发布明细可区分账号。 |

### P2（可选）

| 编号 | 描述 | 验收标准 |
| --- | --- | --- |
| R-12 | **AppSecret 测试连通性**：配置页提供「测试连接」按钮，后端用该账号凭证获取 `access_token` 验证配置正确。 | 返回成功/失败及错误原因，不泄露明文。 |
| R-13 | **账号数量上限提示**：系统对账号数设软上限（如 20）并提示。 | 超过上限时阻止新增并提示。 |
| R-14 | **操作审计日志**：账号的增删改记录操作时间/来源。 | 可查询最近变更。 |
| R-15 | **配置页权限控制**：配置页发布类接口接入现有 `API_BEARER_TOKEN` 鉴权。 | 未携带合法 token 无法读写账号配置。 |

**字段说明表**

| 字段 | 类型/控件 | 校验 | 安全约束 |
| --- | --- | --- | --- |
| 账号名称 | 文本输入 | 必填，≤50 字 | 无敏感信息 |
| AppID | 文本输入 | 必填，`wx` 开头，唯一 | 列表脱敏展示 |
| AppSecret | 密码框（`type=password`） | 必填（新增时）；编辑时为空表示不修改 | **明文绝不回传前端**；保存即加密入库 |
| 备注 | 文本输入 | 可选，≤200 字 | — |
| 设为默认 | 开关 | 与 `is_default` 对应 | — |
| 启用 | 开关 | 与 `is_active` 对应 | 禁用账号不进发布下拉 |

### 4.2 发布页改造（`PublishPage.vue`）

在选择「微信公众号」平台卡片后，平台卡片下方新增下拉：

```
┌──────────────────────────────────────────────────────┐
│  选择发布平台                                           │
│  [ 小红书 ]   [ 微信公众号 (选中) ]                     │
│                                                        │
│  目标公众号： [ 北京房源主号 (默认) ▾ ]   ← 新增下拉    │
│              （仅列出 is_active=true 的账号）           │
└──────────────────────────────────────────────────────┘
        [ 发布到微信公众号 ]   ← 带上 wechat_account_id
```

选中值随 `POST /publish/wechat` 的 `wechat_account_id` 字段提交；若用户未选且存在默认账号，前端默认选中默认账号并提示。

## 5. 待确认问题

> 以下问题需在开发前与**用户 / 架构师**澄清，其中 **Q1（存储方案）** 与 **Q2（加密方式）** 为关键决策点，请架构师重点评估。

| 编号 | 问题 | 建议方向 / 影响 |
| --- | --- | --- |
| **Q1** | **多账号存储方案：数据库表 vs 配置文件？** | 建议新增数据库表 `wechat_accounts`（项目已用 SQLAlchemy + aiosqlite，表结构见 §3 R-01），可天然支持 UI 的增删改查与状态管理。若用配置文件则需自行实现读写与并发安全，且不符合"配置页面管理"的产品诉求。**需架构师确认是否新增表、迁移策略（现有 .env 单账号是否作为首个种子数据导入）。** |
| **Q2** | **AppSecret 加密方式？** | 建议后端使用对称加密（如 `cryptography` 的 **Fernet**），密钥从环境变量 `ENCRYPTION_KEY` 读取；或接入 KMS/HashiCorp Vault。明文仅在调用微信 API 前解密。**需架构师评估加密库选型、密钥管理与轮换策略。** 底线：明文绝不落库、绝不回传前端。 |
| Q3 | **发布默认账号逻辑？** | 未指定 `wechat_account_id` 时回落到 `is_default=true` 账号；是否支持"记忆上次选择"？ |
| Q4 | **是否支持账号启用/禁用？** | 建议支持（R-09）；禁用账号从发布下拉剔除，显式指定时后端报错。 |
| Q5 | **`WechatService` 改造方式？** | 建议保留类、改为按账号构造（如 `WechatService.for_account(account)` 或 `create_draft(..., account=...)`），并让 `access_token` 缓存按 `appid` 维度隔离（现 `self._access_token` 为实例级，多账号下会冲突）。 |
| Q6 | **旧单账号 .env 配置如何兼容？** | `config.py` 的 `WECHAT_APPID/WECHAT_APPSECRET` 是否保留为兜底/种子账号？建议保留并作为首个默认账号迁移，避免破坏现有部署。 |
| Q7 | **账号数量上限？** | 微信无硬限制；系统是否设软上限（如 20，见 R-13）？ |
| Q8 | **配置页权限？** | 是否接入现有 `API_BEARER_TOKEN` 鉴权（R-15）？ |
| Q9 | **发布记录关联账号** | `PublishLog` 是否新增 `wechat_account_id` 列（R-11），以便日志追溯具体账号？ |
